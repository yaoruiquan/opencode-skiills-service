# OpenCode Skills 任务控制台使用说明

适用对象：使用前端创建、运行、查看 OpenCode Skills 服务化任务的操作人员。

当前前端是纯静态中文控制台，只调用 `skills-api`，不直接访问 OpenCode Server。任务输入、输出、日志统一落在后端 job 目录中。

## 1. 访问地址

本地环境：

```text
http://127.0.0.1:4101
```

服务器环境：

```text
http://10.50.10.29:4101
```

对应 API 地址：

| 环境 | API 地址 |
|------|----------|
| 本地 | `http://127.0.0.1:4100` |
| 服务器 | `http://10.50.10.29:4100` |

打开页面后，先确认右上角 **API 地址** 是否正确，再点击 **保存**。保存后页面会重新读取后端模板列表。

## 2. 页面区域说明

页面分为四个主要区域：

| 区域 | 用途 |
|------|------|
| 顶部栏 | 设置 API 地址 |
| 左侧工作区 | 选择模板、填写中文配置表单、上传材料、运行任务 |
| 右侧任务区 | 查看当前任务状态、任务进度、历史任务列表和中断按钮 |
| 底部结果区 | 按类型查看输出文件、标准输出、错误输出 |

## 3. 标准操作流程

```text
选择模板 -> 填写标题 -> 新建任务 -> 准备输入 -> 填写模板配置 -> 运行任务 -> 查看输出
```

### 3.1 选择模板

在 **任务模板** 下拉框选择要运行的模板。

选择模板后，页面会自动显示：

- 模板说明
- 建议输入
- 输出约定
- 可配置项
- 执行模式

### 3.2 新建任务

填写 **任务标题** 后点击 **新建任务**。

创建后右侧任务列表会出现一条 job，任务 ID 形如：

```text
job_528e8732674d4071b2e950c3ecdbf34e
```

同一模板下，如果当前任务不是运行中，前端会优先复用当前任务。需要完全隔离输入时，请手动点击 **新建任务**。

### 3.3 准备输入

不同模板有不同输入方式。

Markdown 类模板：

- `md2wechat`
- `custom`

操作方式：

1. 在 Markdown 编辑区直接粘贴内容，或点击 **选择 Markdown** 导入 `.md` 文件。
2. 点击 **保存 article.md**。
3. 点击 **运行任务**。

材料类模板：

- `vulnerability-alert-processor`
- `phase1-material-processor`
- `msrc-vulnerability-report`
- `cnvd-weekly-db-update`
- `phase2-cnvd-report`
- `phase2-cnnvd-report`
- `phase2-ncc-report`

操作方式：

1. 点击 **上传材料目录** 选择整个材料目录。
2. 如需补充零散文件，点击 **追加文件**。
3. 选择 **执行模式**。
4. 填写 **任务备注**，例如指定 DAS 编号或批次目录。
5. 点击 **保存材料** 或直接点击 **运行当前模板**。

材料文件会被上传到当前 job 的：

```text
input/materials/
```

不要在备注里填写本机绝对路径，例如 `/Users/yao/...` 或 `~/Downloads/...`。需要处理的文件必须通过前端上传。

## 4. 模板配置

非自定义模板会显示 **模板配置** 表单。

运行任务时，前端会把表单配置传给后端，后端写入：

```text
input/service-config.json
```

skill 执行时以这份文件作为服务化配置来源。

页面默认使用中文字段表单。需要调试时可以展开 **高级 JSON**，直接查看或编辑原始配置对象，例如：

```json
{
  "das_id": "DAS-T105966",
  "target_path": "",
  "submit": false,
  "dingtalk_notify": false
}
```

点击 **恢复默认** 可以恢复当前模板的默认配置。

重要约束：

- `submit` 默认必须保持 `false`，确认浏览器登录态、验证码处理和材料无误后才改为 `true`。
- `publish` 默认必须保持 `false`，确认发布服务器和密钥可用后才改为 `true`。
- `dingtalk_notify` 默认必须保持 `false`，确认服务器已配置 webhook 后才改为 `true`。
- API Key、Webhook、SSH 密钥、密码等敏感信息不要写入前端配置。

如果开启 `submit`、`publish`、`dingtalk_notify`、`wechat_draft` 或把 `dry_run` 改为 `false`，前端会在运行前二次确认。

## 5. 模板使用说明

### 5.1 公众号转换

模板名：

```text
md2wechat
```

用途：

将 Markdown 漏洞预警文章转换为微信公众号正文 HTML，并生成封面图。

输入：

```text
input/article.md
```

输出：

```text
output/wechat-article.html
output/wechat-cover.png
```

操作：

1. 选择 **公众号转换**。
2. 粘贴或导入 Markdown。
3. 点击 **运行任务**。
4. 任务成功后在输出文件区下载 HTML 和封面。

### 5.2 自定义

模板名：

```text
custom
```

用途：

临时运行自定义 OpenCode prompt，用于 smoke test 或一次性任务。

操作：

1. 选择 **自定义**。
2. 填写 Markdown 或输入文件。
3. 在 **自定义提示词** 中写清楚要做什么。
4. 点击 **运行任务**。

注意：自定义模板没有固定输出契约，结果是否可用取决于提示词。

### 5.3 漏洞预警材料

模板名：

```text
vulnerability-alert-processor
```

用途：

生成漏洞预警 Markdown、Word、PDF 和渲染上下文。

常用配置：

```json
{
  "cve": "",
  "advisory_url": "",
  "vuln_title": "",
  "wechat_draft": false,
  "publish": false
}
```

推荐输入：

```text
materials/
vuln-data.json
downloaded-template.docx
task.md
```

常见模式：

| 模式 | 用途 |
|------|------|
| `full` | 完整流程 |
| `report-only` | 已有结构化数据和模板时只生成报告 |
| `browser-template` | 浏览器模板阶段 |

预期输出：

```text
summary.txt
final.md
final.docx
final.pdf
render_context.json
```

### 5.4 材料整理

模板名：

```text
phase1-material-processor
```

用途：

整理监管上报前材料，重命名批次目录，修改 CNVD/CNNVD docx 模板。

常用配置：

```json
{
  "batch_dir": "2026-03-30-161902(1)",
  "das_id": "",
  "submitter": "恒脑AI代码审计智能体"
}
```

推荐输入：

```text
materials/{批次目录}/DAS-T*/
```

常见模式：

| 模式 | 用途 |
|------|------|
| `batch` | 处理整个批次 |
| `single` | 处理单个 DAS |
| `list` | 列出可处理材料 |

预期输出：

```text
summary.txt
material-result.json
processed-materials/
```

### 5.5 MSRC 预警报告

模板名：

```text

```

用途：

处理 MSRC 安全更新材料包，生成 `report.md`、Word、PDF，并可选发布预览和钉钉通知。

```json
  "publish": false,
  "dingtalk_notify": false
}
```

推荐输入：

```text

```

常见模式：

| 模式 | 用途 |
|------|------|
| `generate` | 生成报告 |
| `format-only` | 仅格式化已有 Word |
| `publish` | 发布预览和通知 |

预期输出：

```text
summary.txt

preview.html
```

### 5.6 CNVD 周库更新

模板名：

```text
cnvd-weekly-db-update
```

用途：

处理 CNVD 每周 XML 数据库更新。

常用配置：

```json
{
  "remote_host": "",
  "remote_user": "root",
  "docker_container": "crawlab",
  "dry_run": true,
  "dingtalk_notify": false
}
```

推荐输入：

```text

task.md
```

常见模式：

| 模式 | 用途 |
|------|------|
| `check` | 只检查 XML 和远端环境 |
| `update` | 执行真实更新 |

注意：

- `dry_run` 默认为 `true`。
- 真实更新前必须确认服务器 SSH、目标容器和 XML 文件都正确。

预期输出：

```text
summary.txt
update-result.json
```

## 6. 任务状态说明

| 状态 | 含义 | 操作建议 |
|------|------|----------|
| `created` | 已创建，未运行 | 上传输入或点击运行 |
| `running` | 正在执行 | 等待，必要时查看标准输出 |
| `retrying` | 模型 capacity 等原因触发重试 | 等待重试完成 |
| `succeeded` | 执行成功，输出契约满足 | 下载输出文件 |
| `failed` | 执行失败 | 查看错误输出和 `summary.txt` |
| `canceled` | 已手动中断 | 查看错误输出确认中断时间 |

右侧状态卡会显示任务进度：

- 任务已创建
- 输入已准备
- 已进入执行队列
- 模型执行
- 完成、失败或已中断

运行中或重试中的任务会显示 **中断任务** 按钮。点击后前端会调用：

```text
POST /jobs/{job_id}/cancel
```

中断请求会终止当前 `opencode run` 子进程，并把 job 状态标记为 `canceled`。

## 7. 查看输出和日志

任务运行后，底部 **输出文件** 区域会按类型列出 `output/` 下的文件。

当前分组包括：

- 执行摘要
- 报告与表格
- 结构化 JSON
- 图片
- 压缩包
- 视频材料
- HTML 预览
- 处理后材料
- 其他文件

点击文件名可以下载。

日志分为两类：

| 日志 | 说明 |
|------|------|
| 标准输出 | OpenCode run 的 JSONL 输出 |
| 错误输出 | OpenCode 或输出契约校验失败信息 |

如果任务失败，优先看：

1. **错误输出**
2. `output/summary.txt`
3. 当前任务状态里的退出码

## 8. 常见问题

### 8.1 提示 `not_found`

通常是访问了不存在的 job、接口路径或输出文件。

处理方式：

- 点击 **刷新当前任务**。
- 确认右侧选中的 job 是否存在。
- 确认 API 地址是否指向正确环境。

### 8.2 模板配置 JSON 无效

前端会阻止运行，并提示 JSON 解析错误。

处理方式：

- 确认使用英文双引号。
- 确认末尾没有多余逗号。
- 确认最外层是对象 `{}`，不是数组 `[]`。

### 8.3 任务一直运行中

复杂 skill 会调用 OpenCode、模型、脚本、浏览器，运行时间可能较长。

处理方式：

- 先等待 1 到 3 分钟。
- 查看标准输出是否仍在更新。
- 如果长时间无变化，再查看错误输出或服务器容器日志。

### 8.4 平台上报没有提交

默认行为是安全模式：

```json
{
  "submit": false
}
```

此时只准备 `form_context.json` 和检查环境，不会提交 CNVD/CNNVD/NCC。

需要真实提交时必须手动改为：

```json
{
  "submit": true
}
```

并确认：

- Docker Chrome 已登录对应平台。
- 材料路径和 DAS 编号正确。
- 验证码或人工确认流程可处理。

### 8.5 输出文件不完整

后端会检查关键输出契约。缺少必需输出时，任务会标记为 `failed`。

处理方式：

- 查看错误输出中的 `missing required outputs`。
- 查看 `summary.txt` 中的失败原因。
- 检查输入材料是否符合模板要求。

## 9. 操作建议

- 每个真实业务批次单独新建 job。
- 不要在一个 job 里混用多个模板。
- 复杂上报任务先用 `submit=false` 跑通表单上下文，再切换到 `submit=true`。
- 前端配置只放业务参数，不放密钥。
- 失败 job 不要直接覆盖输入，建议新建 job 重新跑。
