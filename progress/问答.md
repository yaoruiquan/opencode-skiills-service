# 项目问答

## Q1：目录结构合理吗？

合理的点：
- **三层架构清晰**：frontend → skills-api → OpenCode Server → DeepSeek API，每层职责单一
- **Docker Compose 编排完整**：5 个服务网络/卷/依赖配置清晰
- **Job 目录约定好**：`/data/work/jobs/{id}/{input,output,logs,job.json}` 路径明确
- **配置与代码分离**：`.env` / `config/` 目录

可改进的点：
- `backend/server.js` 841 行单文件膨胀，应拆为 routes/templates/prompts/jobs/runner
- `frontend/server.js` 与 `backend/server.js` 重名易混淆
- 测试文件散落在源码旁，建议集中到 `tests/` 目录
- `progress/` 命名含义模糊，建议改 `docs/` 或改用 GitHub Issues
- 缺少 `package.json` 声明项目元信息
- `data/input` 空目录未被任何服务使用

## Q2：Skills 迁移的原理是什么？

**Skills 在宿主机上开发维护，通过 Docker bind mount 注入容器供 OpenCode 运行，skills-api 不直接执行 skill 逻辑，只负责编排。**

```
宿主机 ~/.claude/skills/   →  bind mount  →  容器 /root/.agents/skills/:ro
   md2wechat/                                     md2wechat/
   phase2-cnvd-report/                             phase2-cnvd-report/
```

- 改 skill 文档/脚本 → bind mount 实时生效，新任务读到新内容
- 改依赖 → 需要 rebuild 运行镜像
- 加新 skill → mount 后 OpenCode 能直接读到，但要同步改 skills-api 模板注册
- 改浏览器/MCP 配置 → 需要重启 opencode-server 容器

服务器迁移只需改 `.env` 中的 `SKILLS_HOST_DIR` 路径（macOS → `/opt/opencode-skills/skills`），容器内部路径不变。

## Q3：Docker 起了什么作用？

### 1. 环境打包
OpenCode + DeepSeek provider + chrome-devtools-mcp + python3 全部在 Dockerfile 装好，本地无需安装任何依赖。

### 2. 网络连通
容器间通过 Docker internal DNS（service name）互访，无 `localhost` 问题。

### 3. Chrome 隔离
三个 browser 服务同镜像不同 profile，登录态/cookie/缓存互不干扰，通过 `docker-browser` profile 开关控制。

### 4. 可移植
`docker-compose.yml` 不变，换环境只改 `.env` 路径。

**一句话**：Docker 把「本地开发的 skills + Chrome profile + OpenCode」打包成可移植的服务单元，skills 从宿主机 bind mount 注入，改 skills 不用 rebuild 容器。

## Q4：本地 skills git 仓库如何更新并同步到本项目？

skills 独立在本项目之外（`~/.claude/skills/`），不和本项目代码混在同一个仓库。同步方式取决于环境：

### 本地开发

```bash
# 1. 进入 skills 仓库目录
cd ~/.claude/skills
git pull

# 2. 不需要重启容器
#    bind mount 让 opencode-server 容器内 /root/.agents/skills/ 实时生效
#    下次 POST /jobs/{id}/run 时 OpenCode 读到的就是新版本
```

### 服务器部署

```bash
# 方案 A：skills 独立 git 仓库
ssh server
cd /opt/opencode-skills/skills
git pull
#   → 如果只改了脚本/文档，无需重启
#   → 如果改了依赖，需 rebuild 运行镜像

# 方案 B：本机 rsync 推送
rsync -av --delete --exclude='.DS_Store' --exclude='node_modules/' \
  ~/.claude/skills/ user@server:/opt/opencode-skills/skills/
```

### 注意事项

| 变更类型 | 是否需重启 | 原因 |
|----------|-----------|------|
| 改 skill 脚本、SKILL.md、references | 否 | bind mount 实时同步，下次 OpenCode run 读到新内容 |
| 改 skill 依赖（如加 python 包） | 是，需 rebuild | 容器内环境不包含新依赖 |
| 新增 skill 目录 | 否 | OpenCode 能读到，但要同步改本项目 templates 注册 |
| 改 OpenCode 配置 / MCP 地址 | 是，需重启 opencode-server | 配置在容器启动时渲染生成 |

### 关键原则

**Skills 是本项目的"插件"，不是代码的一部分。** 两者解耦：

- skills 仓库只管 skill 本身的脚本和文档
- 本项目仓库只管服务化编排（API / 模板 / prompt / 部署）
- 新增一个 skill 要同步做的事情：
  1. skills 仓库：创建 `{skill-name}/`
  2. 本项目：`backend/server.js` 的 `TEMPLATES` 加一条配置
  3. 本项目：`backend/server.test.js` 加对应测试

## Q5：配置文件迁移到服务器后怎么管理？人为修改在哪里改？

配置文件分三个层级，各自有明确的管理方式：

### 层级一：本项目配置（随服务项目仓库）

```
opencode-skills-service/
├── .env                  # gitignore，服务器上手动创建
├── docker-compose.yml    # 入库，版本化管理
├── config/
│   └── opencode.template.json  # 入库，版本化管理
```

| 文件 | 服务器上在哪改 | 怎么同步 |
|------|--------------|---------|
| `.env` | 服务器 `/opt/opencode-skills/.env` | 手动编辑，不提交。含 API key、端口、路径等环境差异 |
| `docker-compose.yml` | 本地改 → git push → 服务器 git pull | 随项目版本管理 |
| `opencode.template.json` | 本地改 → git push → 服务器 git pull | 随项目版本管理，改后需重启 opencode-server |

### 层级二：Skills 仓库配置（随 skills Git 仓库）

每个 skill 目录下可能有自己的配置文件：

```
~/.claude/skills/{skill-name}/
├── SKILL.md
├── .env.example    # skill 自身的环境变量模板
├── config/
│   └── settings.json
└── references/
```

| 文件 | 服务器上在哪改 | 怎么同步 |
|------|--------------|---------|
| `.env` 等 skill 配置 | 服务器 `/opt/opencode-skills/skills/{name}/` | 手动编辑或 git pull。改脚本/文档类配置无需重启 |
| 脚本类文件 | 本地 skills 仓库改 → git push → 服务器 git pull | bind mount 实时生效 |

### 层级三：容器运行时配置（不可直接编辑）

```
# 由 opencode-server/start-opencode.sh 在容器启动时渲染生成
# 写入 Docker volume opencode-config，不落地到宿主机文件系统
```

| 配置 | 生成方式 | 想改怎么办 |
|------|---------|-----------|
| OpenCode 运行时 `opencode.json` | 启动时从模板 + `.env` 渲染 | 改 `.env` 或 `config/opencode.template.json`，然后重启容器 |
| Chrome profile 登录态 | 首次手动登录后保存在 `data/docker-chrome-profiles/` | 直接操作容器内 Chrome 登录，或替换 profile 目录 |
| Job 运行数据 | 每次 run 自动生成在 `data/work/jobs/` | 不要手动修改，通过 API 读取 |

### 总结：谁改什么

```
你要改什么                    →  去哪个文件改
─────────────────────────────────────────────────
API key / 模型 / 端口         →  服务器的 .env
OpenCode provider / MCP 地址  →  config/opencode.template.json + .env
Skill 脚本 / 提示词           →  skills Git 仓库
浏览器 profile / 登录态       →  容器内 Chrome 操作或替换目录
添加/修改/删除模板            →  本项目 backend/server.js
Docker 网络 / 卷 / 端口映射    →  docker-compose.yml
```

## Q6：开发这个项目会遇到哪些难题？怎么解决的？

### 难题 1：Chrome DevTools WebSocket Host 头校验

Chrome DevTools Protocol 对 WebSocket 请求的 Host 头有严格校验，用 `http://browser-cnvd:9332` 直连会被 Chrome 拒绝连接。

**解决**：在 browser-service 容器内加一层 nginx 反向代理，把 `0.0.0.0:933x` 代理到 `127.0.0.1:9222`，同时改写 Host 头为 loopback 地址。

### 难题 2：DeepSeek API capacity 错误

DeepSeek 高峰期频繁返回 "model is at capacity"，任务直接失败。

**解决**：在 skills-api 中实现 attempt plan——首选模型重试 N 次后自动切换到 fallback 模型列表，每次 attempt 独立记录日志。

### 难题 3：macOS 路径 vs 容器路径冲突

本地习惯用 `/Users/yao/xxx`，但容器路径是 `/data/work/...`，prompt 里一出现 macOS 路径就跑不通。

**解决**：强制约束——所有输入走 job input 目录，所有输出走 job output 目录，prompt 中只允许引用容器路径。API 层做 path traversal 校验拦截绝对路径和 `..`。

### 难题 4：skills 在宿主机、OpenCode 在容器，怎么互通

Skills 目录 (`~/.claude/skills/`) 在宿主机，但 OpenCode 在容器里运行，读不到宿主机文件。

**解决**：Docker bind mount 把宿主机 skills 目录以只读方式挂到容器内固定路径 `/root/.agents/skills:ro`。开发时改技能文件即时生效，不需要 rebuild 容器。

### 难题 5：OpenCode 配置随环境变化

本地用 DeepSeek，服务器可能用不同 provider；本地 Chrome 连 `192.168.65.254`，服务器连 Docker service name。

**解决**：`config/opencode.template.json` 放占位符 `${VAR}`，`start-opencode.sh` 容器启动时读取 `.env` 渲染生成真实配置，写入 Docker volume。

### 难题 6：容器重启后 Chrome profile 锁死

Chromium 退出后留下 `SingletonLock`/`SingletonCookie` 等 stale lock 文件，下次启动报错。

**解决**：`browser-service/start.sh` 启动前清理 profile 目录下所有 `Singleton*` 文件。

### 难题 7：OpenCode run 耗时不可控

一个 task 可能跑几分钟甚至十几分钟，API 不能同步等待。

**解决**：`POST /jobs/{id}/run` 立即返回 202，用异步 `runJobAttempts()` 在后台执行，调用方通过轮询 `GET /jobs/{id}` 获取最终状态。

## Q7：怎么给新人讲这个项目的开发思路？

按这个顺序讲，从为什么到怎么做：

### Step 1：先讲项目要解决什么问题

> 我们有一堆本地 skills（漏洞上报、公众号转换、材料整理），以前只能在本机跑 Claude Code 手动执行。现在要把它们封装成 API 和网页，让任何人都能调用，并且能在服务器上跑。

### Step 2：讲整体链路

画这个图：

```
前端 → skills-api → OpenCode Server → DeepSeek API
                      ↓
               skills 目录 + Docker Chrome
```

强调**核心是 skills-api**，它不干活只做编排。

### Step 3：讲 Job 模型

```
/data/work/jobs/{job_id}/
  input/   材料放这里
  output/  结果出到这里
  logs/    日志在这里
  job.json 任务状态
```

这是整个项目的**核心抽象**——所有 skill 的输入输出都统一成这个模型。

### Step 4：讲 Skills 怎么接进来

Skills 不在项目仓库里，通过 bind mount 挂进容器。项目只负责注册模板、生成 prompt，不负责 skill 内部逻辑。

```js
TEMPLATES = {
  md2wechat: {
    name: "md2wechat",
    skill: "md2wechat",   // 对应容器内的 skill 目录名
    inputMode: "markdown",
    requiredInputs: ["article.md"],
  },
};
```

### Step 5：讲 Docker 解决的核心问题

1. **环境打包**——不用在宿主机装 OpenCode / python / chrome-devtools-mcp
2. **网络连通**——容器间 DNS 互访，无 localhost 问题
3. **Chrome 隔离**——三路 Chrome 同镜像不同 profile
4. **可移植**——本地和服务器共用同一套 docker-compose

### Step 6：讲配置分层

```
.env  → 环境差异（API key / 端口 / 路径）
docker-compose.yml → 服务编排（入库版本化管理）
config/opencode.template.json → OpenCode 配置模板
skills/.env  → skill 自身配置
```

### Step 7：讲开发顺序建议

1. 先在本地跑通 `md2wechat` 完整流程，理解 Job 生命周期
2. 加一个新 skill 模板，理解 TEMPLATES 注册和 prompt 生成
3. 理解三路 Chrome 如何工作，跑通一次 CNVD 上报
4. 再考虑鉴权、队列、生产化部署

### 一句话总结

> **skills-api 是路由层只做编排，OpenCode 是执行引擎，skills 是插件通过 bind mount 注入，Docker 打包一切让本地和服务器的行为一致。**

## Q8：为什么要多一个 skills-api，不直接对接 OpenCode Server？

直接对接 OpenCode Server 的话，调用方需要：

- 自己管理 job 目录结构
- 自己处理模板逻辑和 prompt 生成
- 自己处理 DeepSeek capacity 重试和 fallback
- 自己校验输入路径安全
- 自己记录日志和状态
- 不相关的细节（OpenCode 内部 API 格式）也暴露了

**skills-api 做的事情就是把这些"公共脏活"包起来**，让上层（前端或 API 调用方）只需关心：选什么模板、传什么材料。

### 类比

```
OpenCode Server ≈ 数据库      （能干所有事，但接口是 SQL）
skills-api      ≈ ORM/服务层   （封装了常用操作，对外提供简单的业务接口）
```

skills-api 只约 840 行代码，没有引入额外依赖，但把 OpenCode 的调用方式从"你要自己写 opencode run 命令、管理目录、处理错误"变成了"POST /jobs → POST /files → POST /run → GET /jobs/{id}"。

### 如果去掉 skills-api

```text
前端 / 调用方
  → 需要直接了解 OpenCode CLI 参数
  → 需要自己处理目录创建、文件写入
  → 需要自己解析 opencode run 的 stdout/stderr
  → 每个模板的 prompt 要调用方自己拼
  → 每个调用方都要实现一遍 capacity 重试
  → 前端和 OpenCode Server 耦合，改 OpenCode 可能影响前端
```

加上 skills-api 的解耦价值：换执行引擎（比如从 OpenCode 换成别的），前端完全不用改，只需要改 skills-api 的 runner 模块。

## Q9：DeepSeek API 是干什么的？

DeepSeek API 在这个项目里是**模型推理的提供方**。

```
skills-api → OpenCode → DeepSeek Anthropic 兼容 API → DeepSeek 模型
```

具体流程：

1. OpenCode 收到 skills-api 传来的 prompt
2. OpenCode 调用配置中的 provider（`deepseek-anthropic`）
3. provider 向 `https://api.deepseek.com/anthropic` 发送请求
4. DeepSeek 返回模型推理结果（文本、代码等）
5. OpenCode 根据返回内容执行 skill 脚本（渲染文章、填写表单等）

### 为什么用 DeepSeek 而不是 Claude API？

| 因素 | DeepSeek | Claude API |
|------|----------|------------|
| 价格 | 约 1/10 | 标准定价 |
| 国内访问 | 直接通 | 需要代理 |
| Anthropic 兼容 | 原生支持 | 原生 |
| 模型能力 | V4 Flash 足够 skill 场景 | 更强但更贵 |

DeepSeek 的 `deepseek-v4-flash` 在这个项目场景（按 skill 脚本执行固定任务）下性价比更高。如果某个 task 需要更强推理能力，可以通过 `OPENCODE_FALLBACK_MODELS` 配置备用模型。

### 和 skills-api / OpenCode 的关系

```text
skills-api     →  编排（"用 md2wechat 处理 /data/work/jobs/job_xxx"）
OpenCode       →  执行引擎（"读取 prompt，调用 AI，跑脚本"）
DeepSeek API   →  模型推理（"给出指令的执行步骤和脚本调用"）
```

三者分工明确：skills-api 说"做什么"，OpenCode 说"怎么调用 AI 执行"，DeepSeek 说"我来理解并生成执行指令"。

## Q10：浏览器在 docker 里隔离的原理是什么？

一句话：**三个容器实例，同镜像不同 profile，网络和文件系统完全隔离。**

### 1. 进程隔离

每个 browser 服务是一个独立容器，有自己的 PID 命名空间：

```
browser-cnvd  → 容器内 Chromium 进程（PID 1 → chrome ...）
browser-cnnvd → 容器内 Chromium 进程（PID 1 → chrome ...）
browser-ncc   → 容器内 Chromium 进程（PID 1 → chrome ...）
```

容器 A 的 Chrome 崩溃不会影响容器 B。容器 A 的 Chrome 看不到容器 B 的进程。

### 2. Profile 隔离

每个容器挂载不同的宿主机目录作为 Chrome profile：

```yaml
browser-cnvd:
  volumes:
    - ./data/docker-chrome-profiles/cnvd-report:/home/chrome/profile

browser-cnnvd:
  volumes:
    - ./data/docker-chrome-profiles/cnnvd-report:/home/chrome/profile

browser-ncc:
  volumes:
    - ./data/docker-chrome-profiles/ncc-report:/home/chrome/profile
```

效果：

| 维度 | 表现 |
|------|------|
| 登录态 | CNVD 登录 cookie 不会出现在 CNNVD Chrome 里 |
| 缓存 | 各自独立 |
| 收藏夹/历史 | 完全隔离 |
| 插件/配置 | 互不影响 |

### 3. 网络隔离

每个容器有自己的网络命名空间，监听不同端口：

```
browser-cnvd  → 容器内 0.0.0.0:9332 → 宿主机 127.0.0.1:19332
browser-cnnvd → 容器内 0.0.0.0:9333 → 宿主机 127.0.0.1:19333
browser-ncc   → 容器内 0.0.0.0:9334 → 宿主机 127.0.0.1:19334
```

OpenCode 通过 Docker service name + 端口访问：

```
OpenCode → chrome-devtools-mcp → browser-cnvd:9332
                                → browser-cnnvd:9333
                                → browser-ncc:9334
```

### 4. 为什么不用一个容器跑三个 Chrome？

技术上可以一个容器跑三个 Chrome 实例指向不同 profile，但：

| 方案 | 问题 |
|------|------|
| 单容器三 Chrome | 端口冲突、资源竞争、一个崩溃全挂、日志混在一起 |
| 三容器独立 | 完全隔离、独立启停、独立健康检查、可按需 `docker compose up` |

当前方案是最清晰的。每个 browser 独立成 service，通过 `docker-browser` profile 控制是否启动，不跑上报任务时可以省资源。

### 5. 镜像层也隔离

虽然三个服务用同一个 Dockerfile，但构建缓存共享，不占额外空间：

```
browser-service/Dockerfile
  → 构建一次镜像
  → 三个容器共享同一镜像层
  → 只 profile 目录不同（运行时隔离）
```

## Q11：skills 执行是黑盒，还偶尔卡在验证码，怎么解决？

### 黑盒问题

**现状**：只有"运行→成功/失败"，中间状态完全不可见。

**解法**：
- 加 SSE 实时日志流，执行日志实时推送到前端
- 加作业取消接口，卡死了能止损
- prompt 里输出 `[PROGRESS:步骤名]` 标记，前端解析展示执行进度
- 可选：AI 定期截浏览器图，存到 `logs/screenshots/` 供查看

### 验证码卡住问题

**现状**：AI 识别不了验证码就卡死或失败，只能人工去服务器看日志。

**解法**：引入**暂停恢复机制**
1. AI 遇到验证码时写入 `logs/manual_input_required.json`（附截图和说明），然后退出
2. 服务端检测到该文件，将 job 状态设为 `paused`
3. 前端展示暂停原因和截图，用户填写验证码后提交
4. 服务端用用户输入 + 之前已完成步骤重新生成 prompt，继续执行

**加两个 API**：`GET /jobs/{id}/paused-info`（查看暂停详情）、`POST /jobs/{id}/resume`（提交人工输入后恢复）
