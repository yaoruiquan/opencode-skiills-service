# 问题复盘与解决方法

日期：2026-05-14

本文总结 OpenCode Skills Service 服务化过程中已经遇到的主要问题、原因判断、已采取的修复方式，以及后续需要固化的改进点。

## 1. 前端无法访问服务器

### 现象

访问 `http://10.50.10.29:4101` 时浏览器提示连接被拒绝。

### 原因

`frontend` 服务端口最初只绑定在服务器本机回环地址：

```text
127.0.0.1:4101:80
```

外部机器访问 `10.50.10.29:4101` 时无法连接。

### 解决方法

将 `frontend` 端口改为对外监听：

```text
4101:80
```

同时保持 `skills-api` 和 `opencode-server` 只绑定本机：

```text
127.0.0.1:4100:4100
127.0.0.1:4096:4096
```

外部只访问前端，前端 nginx 通过 `/api/` 反代到 `skills-api`。

## 2. Docker Chrome 容器启动后退出

### 现象

`browser-cnvd`、`browser-cnnvd`、`browser-ncc` 容器启动后立刻退出，OpenCode MCP 无法真正连接浏览器。

日志中出现：

```text
chmod: changing permissions of '/home/chrome/profile': Operation not permitted
```

后续还出现过：

```text
chrome_crashpad_handler: --database is required
open() "/dev/stderr" failed (13: Permission denied)
```

### 原因

服务器上 Docker 自动创建的 Chrome profile 挂载目录属于 root。容器原来直接以 `chrome` 用户启动，无法修改 `/home/chrome/profile` 权限，脚本在 `set -e` 下直接退出。

后续调整中还发现：

- `su -m` 保留了 root 的 `HOME`，导致 Chromium crashpad 初始化异常。
- nginx 以普通用户运行时写 `/dev/stderr` 权限不足。

### 解决方法

将 browser 镜像入口改为 root 启动，先修正 profile 目录权限，再降权到 `chrome` 用户运行 Chromium：

- root 阶段执行 `mkdir/chown/chmod`
- `su -s /bin/bash chrome` 进入正确用户环境
- stale `Singleton*` profile lock 做 best-effort 清理
- nginx error log 改写到 `/tmp`
- 增加 Chromium crashpad 相关禁用参数

验证结果：

- 三个 browser 容器均为 `healthy`
- `/json/version` 正常返回
- MCP 显示三路 `connected`

## 3. Chrome DevTools WebSocket 被 403 或连到 127.0.0.1

### 现象

CNNVD/CNVD/NCC skill 调用 MCP 时出现：

```text
Could not connect to Chrome. Cause: connect ECONNREFUSED 127.0.0.1:9222
Rejected an incoming WebSocket connection ...
```

### 原因

Chromium DevTools 默认返回：

```text
ws://127.0.0.1:9222/devtools/...
```

但 OpenCode 容器内访问 Docker Chrome 应该走：

```text
ws://browser-cnnvd:9333/devtools/...
```

同时 Chromium 对 WebSocket Origin 有校验，直接把 Host 改成 Docker 服务名又会触发：

```text
Host header is specified and is not an IP address or localhost
```

### 解决方法

在 browser-service 内用 nginx 代理 DevTools：

- HTTP 请求转发到 `127.0.0.1:9222`
- 保持 `Host: 127.0.0.1:9222`，避免 Chromium 拒绝非 localhost Host
- 使用 `sub_filter` 将 JSON 返回中的 `ws://127.0.0.1:9222` 改写成 `ws://browser-*:933*`
- Chromium 启动参数设置 `--remote-allow-origins=http://browser-*:933*`

验证结果：

```text
browser-cnvd:9332  HTTP/1.1 101 WebSocket Protocol Handshake
browser-cnnvd:9333 HTTP/1.1 101 WebSocket Protocol Handshake
browser-ncc:9334   HTTP/1.1 101 WebSocket Protocol Handshake
```

## 4. 前端上传文件链路和大文件限制

### 现象

需要确认前端上传材料是否真的传到服务器再执行。

### 当前链路

```text
浏览器前端
  -> POST /api/jobs/{job_id}/files
  -> frontend nginx
  -> skills-api:4100
  -> /data/work/jobs/{job_id}/input
  -> OpenCode/skill 读取 input/
```

服务器实际持久化路径：

```text
/var/lib/opencode-skills/work/jobs/{job_id}/input
```

容器内路径：

```text
/data/work/jobs/{job_id}/input
```

### 当前实现特点

前端不是 multipart 上传，而是读取文件后用 JSON 传 base64：

```json
{
  "filename": "materials/...",
  "contentBase64": "..."
}
```

### 风险

base64 会使请求体膨胀约 33%。当前 nginx 和 API 请求体限制均为 200MB，超大文件可能失败。

### 后续建议

保留当前方案用于 PoC，但后续建议改成 multipart/form-data 或分片上传，减少内存占用和请求体膨胀。

## 5. job 状态 completed/succeeded 不统一

### 现象

历史代码中存在 `completed` 和 `succeeded` 两种成功状态，前端展示和 API 判断容易不一致。

### 原因

早期实现遗留了 `completed`，后续又引入 `succeeded`。如果不收敛，前端状态过滤、轮询和结果展示会出现歧义。

### 解决方法

API 返回层统一归一化：

```text
completed -> succeeded
```

并保留对历史 job 的兼容。

## 6. submission-result 已成功但 job 被 canceled 覆盖

### 现象

CNNVD 任务已经提交成功，生成了：

```text
output/submission-result.json
```

但用户随后中断任务，原始 job 状态变成：

```text
canceled
```

前端可能仍显示“已中断”，无法直观看到平台其实已经提交。

### 原因

服务层对 `submission-result.json` 的成功字段识别不完整。实际 CNNVD 结果写的是：

```json
{
  "cnnvdId": "CNNVD-2026-81852337",
  "status": "待研判"
}
```

但旧逻辑主要识别 `cnnvd_id` 和 `success/submitted` 等状态。

### 解决方法

扩展成功结果识别：

- 支持 `cnvdId`、`cnnvdId`、`nccId`
- 支持 `platformId`、`submissionId`
- 将 `待研判`、`待披露` 视为“平台已提交”
- 返回 `effectiveStatus=submitted`、`effectiveStatusLabel=平台已提交`

验证结果：

```text
status: canceled
effectiveStatus: submitted
platformId: CNNVD-2026-81852337
```

## 7. vulnerability-alert-processor 缺少 summary.txt

### 现象

`vulnerability-alert-processor` smoke 中，任务失败或部分成功时没有稳定写出：

```text
output/summary.txt
```

导致前端和 API 无法给出清晰结果。

### 原因

复杂 skill 的产物较多，失败路径没有统一兜底摘要。API 只看结构化产物时，用户只能看到失败状态，不知道卡在哪里。

### 解决方法

增加 fallback summary 机制：无论成功、失败、取消或缺少部分产物，尽量写出 `summary.txt`，包含：

- template
- outcome
- missing outputs
- error
- existing outputs

## 8. CNVD 防火墙/人工识别

### 现象

CNVD 访问过程中遇到防火墙或人工识别页面，自动流程无法继续。

### 原因

这类验证不是普通登录验证码，也不是提交验证码，OCR 脚本无法可靠处理。必须让前端暴露人工动作或等待用户处理。

### 解决方法

增强防火墙/人工验证检测：

- 将 WAF/Cloudflare 类页面识别为 pending human action
- 前端展示历史失败人工验证动作
- 普通登录验证码仍由 skill 内 OCR 脚本处理
- 防火墙人工识别走前端人工处理或明确失败摘要

## 9. CNNVD 附件上传失败

### 现象

CNNVD 第 3 页上传视频和 PoC 时，MCP `upload_file` 返回成功，但页面里仍显示：

```text
No file chosen
fileListAfter: []
fileCount: 0
```

### 原因

CNNVD 页面使用 Vue/Element UI 自定义上传组件，真实 `<input type="file">` 被隐藏。MCP `upload_file` 只能影响浏览器文件输入，不一定能触发 Vue 组件内部状态和自定义 `httpRequest`。

同时，CNNVD 页面是 HTTPS，尝试让页面 fetch job 容器内临时 HTTP server 文件时会遇到混合内容、网络隔离或容器地址不可达问题。

### 解决方法

将 CNNVD 上传逻辑固化为确定性脚本：

```text
phase2-cnnvd-report/scripts/upload_cnnvd_attachments.py
```

流程：

1. 通过 MCP 从页面读取登录 token，只在当前执行上下文使用，不写入 output 或日志。
2. 脚本直接调用 CNNVD 上传接口：

   ```text
   POST https://www.cnnvd.org.cn/web/compatibilityProduct/importImplementImg
   ```

3. 上传字段：

   ```text
   file
   fileField=1  # 视频
   fileField=2  # PoC
   ```

4. 将返回的 server file URL 写入临时日志 JSON。
5. 生成 browser evaluate script，把上传结果回填到 Vue/Element UI 上传组件。

后续 CNNVD 上报默认使用该脚本，不再优先依赖 MCP `upload_file` 或临时 HTTP server。

## 10. OpenCode 临场摸索导致流程不稳定

### 现象

在 CNNVD 上传问题中，OpenCode 依次尝试了 hidden input、DataTransfer、临时 HTTP server、直接 POST 上传接口等方案，耗时较长。

### 原因

skill 文档只说明“上传附件”，没有把已经验证过的平台接口和 Vue 组件回填方式固化为脚本。模型每次执行都需要重新推理和试错。

### 解决方法

将稳定路径写入 skill：

- `SKILL.md` 增加脚本说明
- `references/runtime-rules.md` 增加“附件上传确定性流程”
- 新增 `scripts/upload_cnnvd_attachments.py`

原则：平台上报类任务中，能脚本化的步骤不要让模型临场猜。

## 11. progress.jsonl 更新不完整

### 现象

部分 job 实际已经进入上传、提交、提取编号阶段，但前端进度仍停留在较早阶段，例如“登录成功”。

### 原因

OpenCode 的详细操作写进了 `logs/run.jsonl`，但业务阶段没有持续追加到：

```text
logs/progress.jsonl
```

前端主要依赖 progress 事件展示业务进度，因此看起来像“卡住”。

### 解决方法

已在 prompt 和服务契约里强调每个状态机阶段必须写 progress。后续还需要继续固化：

- 上传开始/完成写 `upload`
- 提交开始/完成写 `submit`
- 提取编号写 `extract_id`
- 最终写 `summary`

## 12. 本地 skills 与服务器 skills 同步

### 现象

修复 skill 后，需要确认是不是先同步本地再同步服务器。

### 结论

是的，正确顺序是：

```text
本地 skills 源目录
  -> 提交 git
  -> rsync 到服务器 /opt/opencode-skills/skills/{skill}
  -> 重启或重新运行 OpenCode job
```

### 当前约定

本地主要源目录：

```text
/Users/yao/.claude/skills
```

服务器运行目录：

```text
/opt/opencode-skills/skills
```

OpenCode 容器内挂载路径：

```text
/root/.agents/skills
```

## 13. 服务器环境与本地环境差异

### 现象

服务器宿主机没有 `node`，脚本校验需要在容器内执行。

还遇到服务器 Python 版本较旧，不支持：

```python
from __future__ import annotations
```

### 解决方法

- Node 校验在容器中执行：

  ```bash
  docker exec opencode-skills-api node --check /app/server.js
  ```

- 新增 skill 脚本避免使用较新 Python 特性，保持对服务器 Python 的兼容。

## 15. 后续建议

1. 将前端上传从 JSON base64 改为 multipart 或分片上传。
2. 将 CNNVD/CNVD/NCC 的附件上传、验证码、提交结果提取继续脚本化。
3. 前端展示优先使用 `effectiveStatus`，不要只看原始 `status`。
4. 对所有 submission 类 skill 统一 `submission-result.json` 字段规范。
5. 对 `progress.jsonl` 做服务层兜底，避免模型漏写阶段时前端看起来卡住。
6. 每次修改 skill 后执行本地 git commit，再同步服务器，避免本地和服务器行为不一致。
